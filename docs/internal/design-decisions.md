# AK Switch 设计决策记录

## 不做的（已评估后排除）

| 方向 | 排除理由 |
|------|----------|
| CD / Release Pipeline | 纯自用，不需要自动发布 |
| 性能优化 | 50 QPS p99 15.8ms 已远超个人需求，瓶颈在上游 rate limit 不在 AK Switch |
| Key 选择策略 | 单 provider 内 Key 间无负载差异 |
| 外部密钥管理（Vault/KMS） | 本地 AES-256-GCM 加密已够安全 |
| 优雅降级（重试队列/半开） | HTTP 代理场景无法暂存请求；Key 切换成本极低，当前熔断器足够 |
| Dashboard 增强 | 已够用，无真实需求 |
| 请求/响应预处理 | ccswitch 已成熟，不重复造轮 |

## 已知约束

- **ccswitch 领域不碰** — 格式化/整流/转发、provider 路由、请求修改、响应变换等 ccswitch 已成熟的功能不重复造轮。详见 `.agents/documents/ccswitch-analysis.md`。
- **WSL2 9p 文件系统不支持 inotify** — 容器内热重载不会触发（不影响裸跑）
- **高并发性能瓶颈** — 100+ QPS 开始饱和，个人场景到不了这个量级

### ✅ 已完成 / 已覆盖

- **项目改名（Alvus → AK Switch）** — 原话：*"关于更改项目名以及整个 IP 的名字，会很麻烦吗？我想改名为 AK switch……包括GitHub仓库的名字以及URL的地址,这些能不能方便地更改"*
  → 结论：PR #41 已合并。Module 名、CLI 名、Metrics 前缀、配置路径全量改名。仓库名需手动在 GitHub 页面修改。

- **彩色日志输出** — 原话：*"关于运行时的日志输出，可不可以添加一些颜色？比如说那些，你懂吧，我们可以来讨论。"*
  → 结论：PR #44 已合并。零依赖 ANSI 彩色 slog Handler，自动 TTY 检测 / NO_COLOR 支持。

- **429 重试机制** — 原话：*"包括关于重试的机制，比如遇到 429 之后的重试机制。我不清楚现在是一个怎样的机制，但我倾向于碰到 429 之后，就不要在短时间内再去重试了，把时间拉长一点。而且，这些时间都是要弄成可以自定义的配置。"*
  → 结论：PR #42 已合并。429 触发长冷却（指数退避），非永久禁用，配合手动 Enable API。

- **每日限额自动解禁** — 原话：*"关于key达到了今日限额之后的长期禁用是会被自动解禁的吗？就类似于隔一段时间检查一次健康，这样就可以避免他恢复额度之后我们又没能及时使用。"*
  → 结论：长冷却到期自动恢复，配合手动 Enable API 可随时解禁。暂无主动健康检查轮询的必要。

- **Key 分配算法好奇** — 原话：*"我对它将每个请求均匀分配到每个 API key 的机制很好奇……这是不是目前的最优解？有没有更好的解法？加了我们这一层 API key 之间的轮转之后，会比直接直连增加多少延迟？"*
  → 结论：研究完成（`.research-temp/`）。当前 round-robin 已是最优解，单 provider 内 Key 间无负载差异。加代理层延迟约 <1ms（本地 loopback）。

- **日志持久化** — 原话：*"崩溃后无法查日志，是否需要持久化？如何与运行时日志共存？"*
  → 结论：PR #47 已合并。lumberjack 文件日志轮转 + crash.log 崩溃恢复，运行时日志和持久化日志双写共存。

- **PID 预检 + TomlConfig 日志字段修复** — 原话：*（启动时发现日志文件不生成，排查发现 TomlConfig 顶层缺失 LogFile 字段；后来又发现多个实例可同时启动，无防重复保护）*
  → 结论：PR #49 已合并。TomlConfig 结构体补齐日志字段并传播到各 provider；启动时读取 PID 文件检查进程存活，防止重复运行。

- **ColorHandler 日志粘合修复** — 原话：*"怎么 provider 之前的字母全部都跟 provider 粘在一起了呀？"*
  → 结论：直接修复。ColorHandler 的 msg 输出后缺少与 attrs 之间的分隔符，导致 `INFO proxy successprovider=sensenova` 粘合。补上空格后正常。
